# 架构原则

- 单一文件行数限制\*\*：任何文件不得超过 1500 行，超出必须拆分。拆分顺序与策略如下：
  - 阶段 1：提取纯函数（无副作用，迁移成本最低）
  - 阶段 2：拆分 UI 组件（保持 Props 接口稳定）
  - 阶段 3：重构业务逻辑（输入输出保持一致）
  - 阶段 4：优化数据层（保持 API 契约不变）
- 导入规范\*\*：使用路径别名 `@/`，禁止相对路径 `../../`
- 导出规范\*\*：每个文件最多一个默认导出，鼓励命名导出
- 不同功能务必拆分为独立函数，严禁混杂；仅当逻辑真正复用时方可提取公共函数

### 重构流程强制要求

对于涉及>2个文件变更的任务：

- 必须使用sequential-thinking模式
- 最少进行3轮思考-确认-执行循环
- 每轮输出当前进度和下一步计划
- 最终输出完整的变更摘要

# 代码编写规范

- 编写任何代码前，先完整阐述方案并等待确认；需求模糊时务必提出澄清。
- 若一次任务需改动 3 个以上文件，立即暂停并拆分为更小的子任务。
- 代码完成后，列出潜在缺陷并给出对应的测试用例建议。
- 发现 Bug 时，先编写可复现的测试，再持续修复直至测试通过。
- 每次被纠正后，将新规则追加至 project_rules.md，杜绝重复犯错。
